# AutoSaving Alarm 提醒闭环升级 Plan

## 1. 当前实现现状

基于当前代码，现版本已经具备以下基础能力：

- 托盘常驻：`TrayAppContext` 负责托盘图标、菜单、启动流程和 1 秒轮询。
- 右下角置顶提醒窗：`ReminderWindow` 已经是 `TopMost = true` 的提醒窗。
- 固定周期调度：`ReminderScheduler` 以 `AnchorTimeUtc + IntervalMinutes` 为周期锚点。
- 已保存入口：托盘菜单和提醒窗都能触发 `AcknowledgeReminder()`。

但当前闭环不符合这次目标，核心差异有两点：

- `我刚保存了` 只会结束当前提醒，不会从当前时刻重新开始计时。
- 一旦进入提醒态，调度器会把当前提醒冻结在 `_currentReminderDueUtc`，因此“连续几次都没确认”的升级链路实际上还不存在。

这意味着：现在更像“固定周期提醒器”，而不是“提醒 -> 确认 -> 重开计时 -> 未确认则逐级升级”的闭环提醒器。

## 2. 本次目标

把现有实现升级为更实用的提醒闭环，目标语义定为：

- 到点后弹出提醒。
- 用户点击 `我已保存` 后：
  - 立即关闭当前提醒态。
  - 是否以当前时刻重新开始计时，由用户配置决定。
- 如果用户连续多个周期都没有确认：
  - 提醒强度逐级升级。
  - 升级方向优先使用当前项目已经具备或易落地的原生能力：置顶保持、窗口闪烁、颜色强化、系统提示音重复、托盘提示强化。

新增一项显式开关：

- `确认后重置周期`
  - 开启：点击 `我已保存` 后，以当前时刻重建下一轮计时
  - 关闭：点击 `我已保存` 后，仅结束当前提醒，继续沿用原周期节奏

默认建议：

- 默认开启 `确认后重置周期`
- 原因是这更符合“提醒闭环”的主目标
- 但保留关闭选项，以兼容当前项目已有的“固定周期提醒”使用习惯

## 3. 目标交互语义

### 3.1 正常闭环（开启重置时）

以 `15` 分钟为例：

- `10:00` 用户完成一次保存，点击 `我已保存`
- 新一轮计时从 `10:00` 开始
- `10:15` 触发提醒
- 用户若在 `10:16` 点击 `我已保存`
- 下一次提醒改为 `10:31`，而不是继续沿用旧锚点

这里的 `我已保存` 不再只是“确认我看到了”，而是“我刚完成了一次保存，请从现在重新计时”。

### 3.2 固定周期模式（关闭重置时）

仍以 `15` 分钟为例：

- `10:00` 开始计时
- `10:15` 触发提醒
- 用户在 `10:16` 点击 `我已保存`
- 下一次仍然是 `10:30`

也就是说，关闭开关后，产品行为回到当前项目已有的固定周期语义，只是补上“连续未确认可升级”的能力。

### 3.3 未确认升级

仍以 `15` 分钟为例：

- `10:15` 首次提醒，未确认
- `10:30` 视为连续第 2 个未确认周期，升级提醒强度
- `10:45` 视为连续第 3 个未确认周期，再次升级

也就是说，未确认次数不按“弹窗出现次数”算，而按“已经跨过了多少个完整提醒周期仍未确认”算。

### 3.4 升级强度建议

为了贴合现有 WinForms + 托盘结构，先按固定三级策略落地：

- Level 1：第 1 个未确认周期
  - 托盘图标进入告警态
  - 弹托盘气泡
  - 显示置顶提醒窗
- Level 2：第 2 个未确认周期
  - 提醒窗颜色更强烈
  - 标题改为更明确文案，例如 `还没保存`
  - 窗口执行闪烁或前景抢占一次
  - 播放一次系统提示音
- Level 3：第 3 个及以上未确认周期
  - 持续保持置顶
  - 提醒窗进入更明显的闪烁节奏
  - 按节流间隔重复播放系统提示音
  - 托盘文案显示 `已连续错过 N 次`

本阶段不建议一开始就引入大量可配置项，先把默认策略做稳。

## 4. 基于现有代码的改造点

## 4.1 ReminderScheduler

这是本次改造的主入口。

当前问题：

- `Evaluate()` 一旦发现提醒触发，会把 `_currentReminderDueUtc` 固定住。
- `Acknowledge()` 目前只有固定周期这一种处理路径，不支持用户选择。

计划改造：

- 去掉“当前提醒被冻结”的设计，让提醒态可以随着周期推进自然升级。
- 增加 `AcknowledgeResetPolicy` 或等价布尔配置，例如 `AcknowledgeResetsCycle`
- `Acknowledge(nowUtc)` 改为按配置分支处理：
  - 若开启重置：
    - `AnchorTimeUtc = nowUtc`
    - `LastAcknowledgedAtUtc = nowUtc`
    - 清空当前未确认提醒状态
  - 若关闭重置：
    - 保留当前 `AnchorTimeUtc`
    - `LastAcknowledgedAtUtc = nowUtc`
    - 仅结束本次提醒态，不改变后续固定周期
- 新增“未确认周期数”的计算：
  - `overdue_cycles = floor((nowUtc - AnchorTimeUtc) / interval)`
  - 仅当 `overdue_cycles >= 1` 时进入提醒态
- `Evaluate()` 输出不再只区分 `Normal / Reminder / Paused`
  - 还要给出 `OverdueCycles`
  - 还要给出 `EscalationLevel`
  - 还要区分“本次刚进入新等级”还是“仍停留在旧等级”

建议把“是否该触发一次性副作用”也纳入快照，例如：

- `ReminderTriggered`
- `EscalationAdvanced`
- `ShouldPlaySound`
- `ShouldFlashWindow`

这样 `TrayAppContext` 只消费快照，不再自己猜状态跳变。

## 4.2 ReminderSnapshot

当前快照字段过少，不够支撑闭环和升级。

建议扩展为至少包含：

- `State`
- `ReminderTriggered`
- `CurrentReminderDueUtc`
- `NextReminderUtc`
- `OverdueCycles`
- `EscalationLevel`
- `EscalationAdvanced`

必要时可再加：

- `CurrentCycleStartedUtc`
- `NextEscalationUtc`

## 4.3 TrayAppContext

当前职责已经集中在这里，适合继续做提醒表现编排。

计划改造：

- `AcknowledgeReminder()` 调用新语义的 `Acknowledge()`，具体是否重建周期由设置决定。
- `RefreshState()` 根据 `EscalationLevel` 控制不同表现。
- 为避免每秒重复触发副作用，需要只在等级变化时做一次动作：
  - 托盘气泡
  - 系统提示音
  - 窗口抢前台/闪烁
- 托盘菜单文案从 `我刚保存了` 调整为 `我已保存`
- 托盘提示文本增加未确认信息，例如：
  - `AutoSavingAlarm - 该保存了`
  - `AutoSavingAlarm - 已连续错过 2 次`
- 可在托盘提示或提醒窗副文案中补一句当前模式，降低歧义：
  - `确认后将重新计时`
  - `确认后保持固定周期`

这里建议把“副作用节流”状态保存在 `TrayAppContext` 运行时，而不是落盘到配置。

## 4.4 ReminderWindow

当前窗体已经可复用，但表现层需要增强。

计划改造：

- 主按钮文案从 `我刚保存了` 调整为 `我已保存`
- 新增一行状态文案：
  - `已连续 1 个周期未确认`
  - `已连续 3 个周期未确认`
- 根据升级等级切换配色和标题：
  - Level 1：`该保存了`
  - Level 2：`还没保存`
  - Level 3：`请立即保存`
- 增加轻量闪烁机制：
  - 可用 `Timer + BackColor/标题栏强调` 实现
  - 不引入自绘复杂度

窗口仍然保持“不是主窗口，只在提醒时出现”的设计，不改产品形态。

## 4.5 AppSettings / SettingsStore / SettingsForm

当前配置模型需要小幅扩展，但仍然不必做成高度可配置。

本阶段建议：

- 保留现有 `IntervalMinutes`
- 保留现有 `StartWithWindows`
- 保留现有 `IsPaused`
- 保留现有 `ResumePolicy`
- 新增 `AcknowledgeResetsCycle`
- `AnchorTimeUtc` 继续保留，但语义调整为“最近一次开始计时的时刻”
- `LastAcknowledgedAtUtc` 保留，用于记录最近一次明确确认保存的时刻

设置页建议新增一项明确开关：

- `确认后重置周期`
  - 勾选时：我已保存 -> 从现在重新计时
  - 不勾选时：我已保存 -> 仅结束当前提醒，保持固定周期

建议把这项放在提醒间隔下面，因为它直接决定核心调度语义。

本阶段先不要新增以下配置，避免设置页过快膨胀：

- 自定义升级阈值
- 自定义每级声音
- 自定义闪烁频率

如需扩展，下一阶段再加一个“提醒强度”分组设置。

## 4.6 README / 产品文案

当前 README 明确写了“确认后不重置周期”，这会与新增可配置开关冲突。

因此开发本功能时，必须同步改以下文案：

- README 首页价值主张
- 核心特性中的“提醒模式”
- FAQ 中“和普通提醒器的区别”
- 设置窗口说明文案
- 提醒窗口按钮文案
- 托盘菜单文案

否则用户会按“固定周期是唯一行为”去理解，实际产品却已经变成双模式。

## 5. 建议实现顺序

### Phase 1：先改调度语义

目标：

- `我已保存` 后是否从当前时刻重新计时可配置
- 两种模式下都能正确算出连续未确认周期数

涉及文件：

- `src/AutoSavingAlarm/Services/ReminderScheduler.cs`
- `src/AutoSavingAlarm/Services/ReminderSnapshot.cs`

### Phase 2：接入升级表现

目标：

- Level 1 / 2 / 3 三档提醒表现落地
- 避免 1 秒 Tick 导致副作用重复触发

涉及文件：

- `src/AutoSavingAlarm/Application/TrayAppContext.cs`
- `src/AutoSavingAlarm/UI/ReminderWindow.cs`

### Phase 3：收口文案与设置

目标：

- 设置页新增 `确认后重置周期` 开关
- 托盘菜单、提醒窗、README 文案统一到“双模式”新语义

涉及文件：

- `src/AutoSavingAlarm/UI/SettingsForm.cs`
- `README.md`

## 6. 验收标准

- 到点出现提醒后，若开启 `确认后重置周期`，点击 `我已保存` 后下一次提醒从点击时刻重新计算。
- 到点出现提醒后，若关闭 `确认后重置周期`，点击 `我已保存` 后下一次提醒仍沿用原固定周期。
- 用户在第 1 次提醒后不处理，跨过下一个完整周期时，提醒强度会升级，而不是一直停留在同一强度。
- 连续未确认 3 个周期后，界面和提示音明显强于首次提醒。
- 用户一旦点击 `我已保存`，未确认次数立即清零，提醒强度恢复为初始级别。
- `暂停提醒` 仍然生效，暂停期间不累计未确认周期。
- 应用退出再启动后，锚点时间和最近确认时间仍可正确恢复，不会出现刚启动就错误升级。
- README、设置页和托盘菜单文案不再保留“确认后不重置周期”的旧描述。

## 7. 本轮明确不做

- 自动检测外部软件是否真的执行了保存
- 全局快捷键
- 自定义升级策略编辑器
- 多套提醒配置
- 云同步

## 8. 结论

这次需求不是在现有弹窗上加一个按钮文案，而是把产品核心语义升级成“提醒确认策略可配置”的闭环提醒器：用户可以选择保留当前固定周期，也可以选择确认后重新计时；同时在“未确认”这条支线上补一套渐进升级机制。

从当前项目结构看，最合适的落点是：

- 用 `ReminderScheduler` 重写提醒状态机
- 用 `TrayAppContext` 承接升级副作用
- 用 `ReminderWindow` 承接更明显的视觉反馈
- 同步改 README 和设置文案，避免产品语义前后冲突
