# AutoSaving Alarm v1.2 体验增强 Plan

## 1. 当前实现基线

基于当前仓库，现版本已经具备以下能力：

- 托盘常驻、单实例、自启动。
- 双模式确认策略：
  - 确认后重新计时。
  - 确认后保持固定周期。
- 连续未确认升级提醒：
  - 托盘告警。
  - 右下角提醒窗。
  - 升级颜色与声音提示。
- 本地 JSON 配置持久化。

这说明项目的 v1 主链路已经成立，下一阶段不需要再推翻结构，而是沿着现有模块继续增强：

- `TrayAppContext` 继续做托盘生命周期和状态编排。
- `ReminderScheduler` 继续做提醒状态机。
- `SettingsStore` 继续做本地配置读写。
- `ReminderWindow` / `SettingsForm` 继续承接用户交互。

当前仍然存在的缺口主要有四类：

- 缺少 `稍后提醒`，用户只能“我已保存”或“暂停”，中间态不够自然。
- 缺少 `工作时段`，目前默认全天都可能提醒，不适合明确上下班边界的使用者。
- 缺少 `空闲检测`，用户离开电脑时仍可能累计提醒或升级强度。
- 缺少 `配置备份恢复`，一旦主配置损坏，目前只能回退默认值，恢复链路不够稳。

## 2. 本轮目标

本轮目标是在不改变产品定位的前提下，把项目从“能用的提醒器”升级为“更贴近真实工作节奏、也更耐长期常驻”的桌面工具。

本轮新增四项能力：

- `稍后提醒`
- `工作时段`
- `空闲检测`
- `配置备份恢复`

本轮仍然坚持以下边界：

- 不接管任何编辑器，不做自动保存。
- 不监听某个具体软件的保存动作。
- 不引入云同步。
- 不把产品做成重型效率平台。

## 3. 目标语义

### 3.1 稍后提醒

`稍后提醒` 的目标不是“当作已保存”，而是“我知道该保存了，但请过几分钟再来提醒我一次”。

建议语义如下：

- 仅在当前处于提醒态时可用。
- 点击 `稍后提醒` 后：
  - 立即关闭当前提醒窗。
  - 暂停当前提醒的视觉与声音副作用。
  - 记录一个 `SnoozeUntilUtc`。
- `稍后提醒` 不等于 `我已保存`：
  - 不写入 `LastAcknowledgedAtUtc`。
  - 不清空原有的逾期背景。
  - 不改变“是否真的保存过”的语义。
- 在 `SnoozeUntilUtc` 之前：
  - 不再弹提醒窗。
  - 不再播放升级提示音。
  - 不再重复托盘气泡。
- 到达 `SnoozeUntilUtc` 后：
  - 重新基于当前真实逾期情况计算提醒等级。
  - 如果用户已经跨过多个完整周期，则直接恢复到对应升级等级，而不是重新回到 Level 1。

本轮建议先用一个简单但完整的版本落地：

- 设置页提供一个 `稍后提醒时长`，候选值先固定为 `5 / 10 / 15 / 30` 分钟。
- 提醒窗与托盘菜单都只提供一个动作：`稍后提醒`。
- 执行时统一使用当前设置里的默认时长。

### 3.2 工作时段

`工作时段` 的目标是让提醒只发生在用户真正希望被打扰的时间段内，而不是全天候运行到深夜。

建议语义如下：

- 用户可以配置：
  - 哪些星期生效。
  - 每天的开始时间。
  - 每天的结束时间。
- 在工作时段之外：
  - 不显示提醒窗。
  - 不弹托盘告警。
  - 不累计未确认升级。
- 从非工作时段进入工作时段时：
  - 默认按“新一轮工作开始”处理。
  - 从当前进入工作时段的时刻重新建立计时锚点。

这里不建议直接复用已有的 `恢复策略` 语义去解释工作时段恢复，因为这会带来一个明显问题：

- 用户晚上下班后长时间不在工作时段。
- 如果第二天继续沿用旧锚点，程序可能一上来就进入高等级逾期状态。

因此本轮建议把工作时段的恢复语义固定为：

- `工作时段重新开始时，默认从当前时刻重新计时`

本轮明确不做：

- 每天多时段配置。
- 节假日日历。
- 轮班模板。

### 3.3 空闲检测

`空闲检测` 的目标是识别“人已经离开电脑”，避免提醒系统在无人操作时无意义地升级或打扰。

建议语义如下：

- 用户可以配置：
  - 是否启用空闲检测。
  - 空闲阈值分钟数。
- 当系统检测到用户连续无输入达到阈值时：
  - 进入 `空闲挂起` 状态。
  - 停止提醒窗、托盘气泡和升级音效。
  - 不继续累计未确认周期。
- 当用户重新回到电脑时：
  - 退出 `空闲挂起`。
  - 从当前恢复时刻重新开始计时。

这里建议把空闲恢复也定义成“恢复即重置”，原因和工作时段类似：

- 用户离开电脑期间没有机会保存。
- 如果回来后立即看到一条高等级逾期提醒，体验会比较差。

与 `稍后提醒` 的区别要明确：

- `稍后提醒` 是用户主动做出的延后决策。
- `空闲挂起` 是系统被动检测到用户不在场。

两者都可以暂时压住提醒，但语义不同，恢复来源也不同。

### 3.4 配置备份恢复

`配置备份恢复` 的目标是提升常驻工具的稳定性，不让一次配置损坏直接把用户打回默认状态。

建议语义如下：

- 主配置仍然是：
  - `%AppData%\\AutoSavingAlarm\\settings.json`
- 同目录自动维护一份最近有效备份，例如：
  - `%AppData%\\AutoSavingAlarm\\settings.backup.json`
- 每次成功保存配置时：
  - 先写临时文件。
  - 校验可反序列化。
  - 再替换主配置。
  - 同步刷新最近有效备份。
- 启动加载时：
  - 优先读取主配置。
  - 若主配置损坏，则自动尝试读取备份配置。
  - 若备份可用，则用备份恢复运行，并给用户一个明确提示。
  - 若主备份都损坏，则回退默认值，但应保留损坏文件副本，方便后续排查。

为了让这个能力对用户可见，建议设置页补一个轻量入口：

- 显示“当前配置文件路径 / 最近备份状态”。
- 提供“恢复最近可用备份”按钮。

本轮先不做：

- 云端备份。
- 多版本历史回滚。
- 跨机器同步。

### 3.5 多状态优先级

加入这四项能力后，提醒状态不再只有 `Normal / Reminder / Paused` 这么简单，必须先明确优先级，否则实现时容易互相打架。

建议优先级如下：

1. 手动暂停。
2. 非工作时段挂起。
3. 空闲挂起。
4. 稍后提醒中。
5. 正常计时 / 提醒升级。

也就是说：

- 如果用户手动暂停，就不再关心工作时段、空闲、稍后提醒。
- 如果当前不在工作时段，就不再进入空闲检测或稍后提醒分支。
- 如果已经进入空闲挂起，也不应再对当前提醒执行额外升级副作用。
- 只有在前面几个抑制条件都不成立时，才进入正常提醒与升级逻辑。

## 4. 基于现有代码的改造点

### 4.1 AppSettings / SettingsStore

本轮配置模型建议新增以下字段：

- `DefaultSnoozeMinutes`
- `SnoozeUntilUtc`
- `WorkScheduleEnabled`
- `WorkdayMask`
- `WorkdayStartLocalTime`
- `WorkdayEndLocalTime`
- `IdleDetectionEnabled`
- `IdleThresholdMinutes`

其中语义建议如下：

- `DefaultSnoozeMinutes`
  - 用户在设置里选定的默认稍后提醒时长。
- `SnoozeUntilUtc`
  - 当前是否处于稍后提醒抑制期。
  - 需要持久化，避免应用重启后丢失稍后状态。
- `WorkdayMask`
  - 用位掩码或等价形式表示周一到周日是否启用。
- `WorkdayStartLocalTime` / `WorkdayEndLocalTime`
  - 保持本地时间语义，不建议转成 UTC 存储后再让用户理解。
- `IdleDetectionEnabled` / `IdleThresholdMinutes`
  - 控制空闲检测是否生效以及阈值。

`SettingsStore` 除字段持久化外，还要承担备份恢复职责：

- 保存流程改成“临时文件 -> 校验 -> 原子替换 -> 刷新备份”。
- 启动加载时，返回值除了 `Settings` 和 `Exists`，最好还能带出恢复来源信息，例如：
  - 正常主配置加载。
  - 从备份恢复。
  - 主备份都失败，回退默认。

这里建议：

- 自动备份恢复仍放在 `SettingsStore` 内部。
- 不额外拆一个很重的配置子系统。

### 4.2 ReminderScheduler / ReminderSnapshot

`ReminderScheduler` 是本轮改造的核心。

本轮建议扩展的能力包括：

- 正常提醒与升级。
- 稍后提醒。
- 工作时段抑制。
- 空闲挂起抑制。

建议扩展状态枚举或等价快照表达，至少覆盖：

- `Normal`
- `Reminder`
- `Paused`
- `Snoozed`
- `SuppressedBySchedule`
- `SuppressedByIdle`

建议快照补充字段：

- `SnoozeUntilUtc`
- `SuppressionReason`
- `IsWithinWorkSchedule`
- `IsIdleSuspended`
- `StateChanged`
- `NeedsUserVisibleRefresh`

计划改造点：

- `Evaluate()` 不再只看“是否到点”。
- `Evaluate()` 先判断状态优先级，再决定是否进入提醒态。
- 新增 `Snooze(nowUtc, minutes)`。
- 新增“从工作时段恢复时重建锚点”的处理入口。
- 新增“从空闲恢复时重建锚点”的处理入口。
- 如果稍后提醒结束时已经积累到更高逾期等级，应直接恢复到真实等级。

### 4.3 新增轻量服务

为了保持现有结构清晰，建议只补最少的独立服务：

- `WorkScheduleEvaluator`
  - 负责判断当前本地时间是否处于工作时段内。
- `UserIdleMonitor`
  - 负责查询系统空闲时长。
  - Windows 下可基于 `GetLastInputInfo` 实现。

配置备份恢复则不建议再拆新模块，先保留在 `SettingsStore` 里即可。

### 4.4 TrayAppContext

`TrayAppContext` 需要承接更多状态编排职责。

本轮计划改造：

- 新增托盘菜单动作：
  - `稍后提醒`
- 根据状态显示不同托盘文案：
  - 正常计时中。
  - 已稍后至某时刻。
  - 当前不在工作时段。
  - 已因空闲自动挂起。
- 只在真正的状态跳变时触发副作用：
  - 托盘气泡。
  - 声音提示。
  - 窗口前置。
- 如果应用是从备份配置恢复启动：
  - 第一次进入稳定状态后，给出一个明确但不过度打扰的提示。

这里要特别避免一个问题：

- 1 秒 Tick 驱动下，同一种抑制状态被重复当成“新事件”。

因此本轮建议把“状态边沿变化”纳入快照或运行时缓存，不要继续只靠当前瞬时状态做判断。

### 4.5 ReminderWindow

提醒窗需要扩展为更完整的“当前周期处理面板”。

本轮计划改造：

- 新增按钮：`稍后提醒`
- 状态文案补充以下信息：
  - 当前逾期等级。
  - 当前确认策略。
  - 若处于稍后提醒恢复后首次弹出，可提示“已从稍后提醒恢复”。
- 如果由于非工作时段或空闲挂起而不应展示，则自动隐藏。

窗口仍然保持：

- 不是主窗口。
- 只在需要提醒时出现。
- 不演变成复杂仪表盘。

### 4.6 SettingsForm

设置页本轮会明显扩容，但仍然应维持“可快速理解”的结构。

建议拆成四组：

1. 提醒基础设置
   - 提醒间隔
   - 确认后重置周期
   - 声音提示
2. 稍后提醒
   - 默认稍后时长
3. 工作节奏控制
   - 启用工作时段
   - 生效星期
   - 开始时间 / 结束时间
   - 启用空闲检测
   - 空闲阈值
4. 配置与启动
   - 开机启动
   - 手动暂停
   - 恢复策略
   - 配置路径 / 备份状态 / 恢复备份

布局上仍然建议：

- 继续使用 `AutoSize + TableLayoutPanel`
- 避免重新退回硬编码像素布局

### 4.7 README / 产品文案

本轮功能加入后，README 和界面说明必须同步更新，重点包括：

- 核心特性区新增：
  - 稍后提醒
  - 工作时段
  - 空闲检测
  - 配置备份恢复
- FAQ 增加：
  - 为什么下班后不提醒了
  - 为什么我离开电脑后没有继续升级提醒
  - 配置损坏后如何恢复
- 设置页文案要降低歧义：
  - 清楚区分 `我已保存`
  - `稍后提醒`
  - `暂停提醒`

## 6. 验收标准

### 6.1 稍后提醒

- 到点提醒时，用户点击 `稍后提醒`，当前提醒窗立即关闭。
- 在 `SnoozeUntilUtc` 到达前，不重复弹出提醒。
- 稍后期间应用退出再启动，稍后状态仍能恢复。
- 稍后结束后，提醒等级按真实逾期情况恢复，不错误重置为 Level 1。

### 6.2 工作时段

- 非工作时段内不弹提醒窗、不响铃、不累计逾期升级。
- 进入工作时段后，从当前时刻重新开始计时。
- 只在勾选的星期范围内生效。
- 时间边界切换时不出现重复提醒或瞬时连弹。

### 6.3 空闲检测

- 用户连续无输入达到阈值后，进入空闲挂起。
- 空闲挂起期间不继续升级提醒。
- 用户恢复输入后，从当前时刻重新开始计时。
- 手动暂停优先级高于空闲挂起，不会互相覆盖出错。

### 6.4 配置备份恢复

- 正常保存配置后，会留下最近一次可用备份。
- 当主配置损坏时，程序能自动回退到备份配置启动。
- 当主备份都损坏时，程序能回退默认配置，但不会静默吞掉损坏文件。
- 用户能在设置页看到当前配置与备份状态。

### 6.5 组合边界

- 稍后提醒期间如果进入非工作时段，返回工作时段后不应错误保留过期的稍后副作用。
- 空闲挂起期间如果刚好跨过工作时段结束，不应在错误时机恢复提醒。
- 手动暂停后再恢复，不应意外恢复到旧的稍后提醒状态。
- 从备份恢复启动后，不应立刻触发错误的高等级逾期提醒。

## 7. 本轮明确不做

- 自动保存文件。
- 绑定 VS Code、Word、Photoshop 等具体软件。
- 节假日日历联动。
- 每天多段工作时段。
- 云同步。
- 全局快捷键。
- 自定义提醒策略编辑器。

## 8. 结论

这一轮不是给现有提醒器再堆几个独立按钮，而是把产品继续往“真实工作节奏感”和“长期稳定常驻”两个方向推进：

- `稍后提醒` 解决中间态操作不自然的问题。
- `工作时段` 解决全天候提醒不合时宜的问题。
- `空闲检测` 解决用户离开电脑时仍被无意义累计提醒的问题。
- `配置备份恢复` 解决桌面常驻工具最怕的配置损坏问题。

从当前项目结构看，这四项功能都可以顺着现有架构自然长出来，不需要重做应用形态。
